- **Tujuan:** Memfasilitasi pertukaran jasa antar mahasiswa (akademik etis & non-akademik).

# ARSITEKTUR SISTEM & LOGIKA BISNIS
Anda harus memahami entitas dan logika berikut saat memberikan saran koding:

## 1. User Roles & Profile (Single Account, Dual Mode)
- User memiliki satu akun tetapi bisa beralih peran (*context switching*):
  - **Seeker:** Posting "Open Projects".
  - **Contributor:** Posting "Service Listings" & Portofolio.
- **Implikasi Koding:** Database schema harus mendukung relasi `one-to-many` untuk services dan projects ke satu user ID yang sama.

## 2. Core Transactional Logic (State Management)
Anda harus menerapkan *State Machine* yang ketat untuk transaksi. Jangan biarkan transaksi melompat status tanpa validasi.

### A. Paid Transactions (Escrow Model)
- Integrasi logika Payment Gateway (Xendit/Midtrans).
- **Alur Logic:**
  1. `PENDING`: Negosiasi.
  2. `AWAITING_PAYMENT`: Seeker checkout.
  3. `ESCROW_HOLD`: Gateway menerima dana (Dana BELUM ke Contributor).
  4. `IN_PROGRESS`: Contributor bekerja.
  5. `VERIFICATION`: Pekerjaan diserahkan.
  6. `COMPLETED`: Seeker approve -> Trigger fungsi pelepasan dana ke Contributor.
  7. `DISPUTED`: Jika ada masalah (admin intervention).

### B. Skill Barter (Cashless)
- **Logika:** `Service A` ditukar dengan `Service B` atau `Project A`. Nilai tukar berdasarkan kesepakatan (mutual agreement).

## 3. Fitur Teknis Lainnya
- **Real-time Chat:** Implementasi websocket/polling untuk negosiasi sebelum transaksi terjadi.
- **Reputation System:** Rating dua arah (Seeker menilai Contributor, dan sebaliknya) yang wajib diisi setelah status `COMPLETED`.
- **Academic Integrity Guardrails:** Saat membuat filter konten atau validasi input, pastikan sistem bisa mendeteksi kata kunci yang mengarah ke kecurangan akademik (misal: "Joki Skripsi") dan memberi flag/blokir.
